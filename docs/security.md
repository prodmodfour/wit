# Wit security model

Wit is designed for a trusted operator on a local, single-user host. Its controls reduce accidental broad media operations, credential disclosure, unsafe state access, and public administrative exposure. They do not turn the host or the media services into a hostile multi-tenant environment.

For vulnerability reporting, see the repository [security policy](../SECURITY.md). For secure settings and file modes, see [configuration](configuration.md). For backup and recovery, see [operations](operations.md).

## Security goals

Wit aims to:

- keep administrative web interfaces on host loopback by default;
- keep credentials out of command arguments, plans, output, logs generated by Wit, examples, and version control;
- make planning non-mutating and require a stored, inspectable plan before mutation;
- require explicit confirmation and fail safely on ambiguity, staleness, mapping inconsistency, or material metadata changes;
- constrain Sonarr mutation to one unmonitored series add, explicit episode monitoring, and one targeted episode search;
- validate service responses and filesystem paths rather than trusting raw data;
- use restrictive, atomic plan persistence;
- avoid shelling out from user-provided CLI values;
- keep tests and CI offline from real media services and credentials.

## Security non-goals and assumptions

The current release assumes:

- the operating-system account running Wit is trusted;
- the Docker administrator and host root are trusted;
- service administrators are trusted to configure users, sources, and remote access correctly;
- the host filesystem and backup destination provide confidentiality appropriate for service databases and media records;
- configured Sonarr and Jellyfin endpoints are the intended services;
- upstream Python packages, image registries, service images, Docker, and the host kernel are part of the supply-chain trust base.

Wit does not provide disk encryption, a secret manager, service account provisioning, API-key scoping, malware scanning, container-to-container isolation, multi-user authorisation, TLS termination, a reverse proxy, public hosting, or media-source validation. A process running as the same operating-system user can generally read that user's configuration and invoke Wit. A Docker or root compromise is outside the protection offered by plan-file permissions.

## Assets

Protect at least the following:

- Sonarr and Jellyfin API keys used by Wit;
- qBittorrent credentials, Sonarr authentication and source credentials, Jellyfin users/API keys, and Seerr upstream credentials in service-owned config;
- service databases and backups, which can contain users, request records, download history, filenames, and library metadata;
- plans and JSON output, which contain show identity and episode-request history even though they contain no secrets;
- the television library and downloads;
- local hostnames, URLs, path layout, service versions, and operational logs;
- the operator's confirmation channel, because `--yes`, `--allow-stale`, and `--allow-mismatch` represent safety decisions.

## Threat model

### Accidental or over-broad mutation

A title may be ambiguous, metadata may have changed, a selector may include the wrong episodes, or automation may repeat a request. Wit addresses these risks with deterministic matching, explicit candidate selection, aired-regular-only selection, complete plan rendering, confirmation, a seven-day age limit, all-coordinate preflight mapping, mismatch reconfirmation, and file/queue idempotence checks.

Residual risks remain: Sonarr must sometimes add a series before episode IDs can be mapped, and the monitor and search calls are not transactional. A failure can leave a newly added series unmonitored or actionable episodes monitored without a successfully submitted search. Operators must inspect the reported failure and Sonarr state rather than assuming rollback occurred.

### Credential disclosure

Credentials could leak through command history, process arguments, permissive files, logs, backups, copied diagnostics, environment inheritance, or a compromised service. Wit never accepts API keys as CLI options and uses redacted secret types and safe errors, but it cannot protect a secret placed in an unsafe environment, shell trace, service log, or backup.

### Malicious or inconsistent remote data

TVmaze and local services return data that cannot be treated as trusted structure. Wit applies strict or bounded Pydantic response models, validates IDs and coordinates, rejects duplicate/inconsistent pagination, bounds requests, strips unsafe control characters at display boundaries, and never evaluates response data as code or a shell command. A compromised authoritative service can still deny operation or return plausible incorrect metadata; plan review and the Sonarr mismatch check are the final human safety layers.

### Local filesystem attacks

Another local process could attempt traversal, symlink substitution, unsafe ownership, partial writes, or plan tampering. Configuration and plan-store code rejects unsafe paths and symlinks, checks ownership and permissions, validates plan IDs and schemas, and uses descriptor-relative, no-follow file access with atomic replacement. These controls do not defend against a process with the same user identity or root privileges.

### Network exposure and service compromise

Administrative services are high-value targets. Compose publishes them only on `127.0.0.1`, but every container joins the same internal and egress bridges and can reach other attached containers. `no-new-privileges` limits one privilege-escalation path; it is not a complete sandbox. A compromised container may access its mounts, peer services reachable on shared networks, and outbound destinations allowed by the host.

### Supply-chain compromise

Compose uses exact release tags rather than floating aliases, and Python dependencies are locked. Tags are not immutable digests, and neither mechanism substitutes for upstream review. Deliberate updates, backups, release-note review, the quality gate, and registry/dependency provenance remain operator responsibilities.

## Trust boundaries and data disclosure

| Boundary | Data crossing it | Security notes |
| --- | --- | --- |
| Operator or Pi → `wit` | show query, selectors, plan ID, confirmations | Pi must use the CLI and must not inspect secret configuration; confirmations must follow display of the exact plan |
| `wit` → TVmaze | title query and selected public TVmaze ID | HTTPS by default; TVmaze can observe requested metadata; no service credential is sent |
| `wit` → Sonarr | API key header, TVDB/series/episode IDs, bounded reads and confirmed mutations | Loopback HTTP by default; Sonarr API keys are powerful and are not provisioned or scoped by Wit |
| `wit` → Jellyfin | dedicated API key, TVDB ID, title/year, bounded catalogue reads | Read-only operations in Wit; the key and endpoint still require protection |
| `wit` → Seerr | credential-free health request | No Seerr login is loaded by Wit |
| Sonarr → qBittorrent | download-client credentials and queue operations | Configured in Sonarr; outside Wit's credential store and direct control |
| Seerr → Sonarr/Jellyfin | service credentials and human request/library data | Configured in Seerr; separate from Wit's plan/apply safety boundary |
| services → host bind mounts | databases, config, downloads, media | Access follows container user IDs and mount modes; Jellyfin's `/tv` mount is read-only |
| `wit` → XDG state | secret-free plan JSON | Private metadata; owner-controlled, atomic storage |

The shared HTTP transport disables redirects and inherited proxy settings (`trust_env=False`) and uses bounded connect/read/write/pool timeouts. This reduces credential forwarding and accidental proxy disclosure. If a remote HTTPS endpoint is configured, ordinary system trust roots are used; certificate pinning is not implemented.

## Secret handling

### Storage locations

| Value | Approved location | Prohibited locations |
| --- | --- | --- |
| Sonarr API key for Wit | owner-only protected TOML, or trusted inherited process environment | CLI arguments, Compose `.env`, plans, issues, committed files |
| Jellyfin API key for Wit | owner-only protected TOML, or trusted inherited process environment | CLI arguments, Compose `.env`, plans, issues, committed files |
| qBittorrent login | qBittorrent service-owned config | Wit config, Compose `.env`, plans, source files |
| Sonarr login/source/download-client credentials | Sonarr service-owned config | Wit config except its API key, shell commands, plans, source files |
| Jellyfin administrator login | Jellyfin service-owned config | Wit config, Compose `.env`, plans, source files |
| Seerr users and upstream credentials | Seerr service-owned config | Wit config, Compose `.env`, plans, source files |

The repository `.env` is intended only for the non-secret data root, user/group IDs, timezone, and host ports. Treat it as machine-private anyway, because local values can reveal topology. Do not place a secret there merely because Git ignores the file.

### Protected TOML

The file selected by `WIT_CONFIG_FILE` must be:

- an absolute path after expansion;
- a regular file, not a symlink;
- owned by the current user; and
- exactly mode `0600` or `0400`.

Unknown fields are rejected, URLs cannot contain embedded credentials, and validation errors omit input values. Environment values override file values. Environment injection is useful for an approved secret manager, but environment values can be inherited by child processes or exposed by same-user debugging facilities. Avoid inline secret assignments that enter shell history, shell tracing, process supervisors with broad read access, or CI logs.

Never ask an operator to paste the file or print the corresponding environment variables. Use the redacted result of `wit doctor` and identify fields by name.

### Rotation and suspected exposure

If a credential may have been disclosed:

1. stop sharing the affected output and preserve only a private, minimal incident record;
2. revoke or rotate the credential in the owning service;
3. update the protected Wit configuration or service-owned integration without putting the value on a command line;
4. restart only if the owning service requires it;
5. run `wit doctor` to validate connectivity;
6. review service access and backup/log destinations for unauthorised use; and
7. report a product vulnerability through [SECURITY.md](../SECURITY.md) if Wit caused the disclosure.

Rotating a Sonarr or Jellyfin API key does not require changing existing plan files because plans contain no credentials.

## Plan and state-file safety

Schema-version-1 plans contain only:

- schema version and random plan ID;
- UTC creation time;
- show title/year plus TVmaze and TVDB IDs;
- selector summary; and
- selected season/episode coordinates and titles.

They reject unknown fields, so accidental secret-like additions are not silently retained. They contain no service headers, cookies, API keys, raw payloads, absolute media paths, or command IDs.

By default, plans live at `${XDG_STATE_HOME:-~/.local/state}/wit/plans/`. The store:

- refuses a relative path, filesystem root, parent traversal, state-directory symlink, or unsafe plan ID;
- creates the state and plan directories with mode `0700`;
- creates plan files with mode `0600` and uses atomic replacement plus filesystem sync;
- uses no-follow, descriptor-relative access;
- requires store directories and files to be owned by the current user and not writable by group or others;
- rejects non-regular files, symlinks, corrupt JSON, unsupported schemas, and filename/content ID mismatches; and
- ignores unrelated entries when listing plan-shaped filenames.

Secret-free does not mean public. Plans reveal viewing intent and should receive owner-only access, private backups, and careful retention. Do not manually edit a plan to bypass staleness, identity, or mismatch safeguards; create a new plan from current metadata instead.

## Apply confirmation and automation safety

A safe automation sequence is:

1. run `wit plan` and show the complete result and plan ID;
2. obtain affirmative confirmation tied to that displayed plan;
3. invoke `wit apply <plan-id> --yes` only after that confirmation; and
4. use `wit status <plan-id>` for observation.

The initial natural-language request, creation of a plan, silence, a standing approval, or `--yes` supplied before review is not confirmation. `--allow-stale` and `--allow-mismatch` approve separate risks and require their own informed decisions. JSON mode does not weaken these rules.

Do not grant Pi a general ability to run raw `curl`, edit service databases, control qBittorrent, or execute arbitrary shell based on a media request. The deterministic CLI is the policy boundary.

## Network exposure

Compose publishes only the four administrative HTTP interfaces, each on host loopback:

- qBittorrent Web UI;
- Sonarr;
- Jellyfin; and
- Seerr.

It does not publish qBittorrent's peer/listening port, Jellyfin discovery ports, a reverse proxy, TLS, or host networking. Loopback binding means another LAN device cannot connect directly. It also means local untrusted processes may still attempt access, so enable each service's authentication and use unique credentials.

Do not change bindings to all interfaces as a shortcut for remote access. A remote design needs, at minimum, an authenticated private network or carefully configured reverse proxy, TLS, service-aware authentication, firewall policy, rate limiting where appropriate, trusted proxy/header settings, and a review of every service's own remote-access guidance. That design is outside this repository and must not weaken the default Compose file.

Container egress is needed for image/service operation and TV metadata/source access, but it is not restricted by destination in Compose. Host firewall and DNS policy are external controls. All four containers share network reachability; the `internal` network name must not be interpreted as hostile workload isolation.

## Container and filesystem posture

Current Compose hardening includes:

- exact application release tags;
- `no-new-privileges` on every service;
- conservative `unless-stopped` restart policies;
- credential-free health checks;
- explicit bind mounts;
- consistent `/downloads` paths for qBittorrent and Sonarr;
- Sonarr-only write access to `/tv`, with Jellyfin read-only access; and
- no Docker socket, privileged mode, host networking, discovery ports, or machine-specific devices.

Limitations include writable container filesystems, no explicit capability-drop policy, common bridge networks, mutable tag provenance, and service config containing powerful credentials. Keep Docker and images patched through the deliberate [update procedure](operations.md#updating), and do not mount additional host paths broadly.

## Logs, output, telemetry, and backups

Wit has no analytics or telemetry subsystem. It sends planning queries to TVmaze and requests to configured local services. Service applications and image registries have their own logging and network behaviour.

Wit errors intentionally omit authentication values and raw payloads. Human and JSON output can still contain show titles, plan IDs, episode coordinates, service versions, and safe operational details. Service logs and autonomous-agent event logs can contain much more. Keep all of them private and share only a redacted minimum.

Backups of `WIT_DATA_ROOT/config` and the protected TOML are secret-bearing. Backups of plans and media are private data. Use encrypted, access-controlled storage; test restore without public exposure; and destroy expired copies according to an operator-approved retention policy rather than ad hoc cleanup. See [operations](operations.md#backup-policy).

## Responsible-use boundaries

Wit is for managing a personal, lawful television library. It does not provide or recommend indexers, trackers, feeds, provider presets, debrid integrations, DRM bypasses, stream-ripping tools, credential extraction, or subscription-service scraping. Operators must configure only sources they are authorised to use and comply with applicable licences, terms, and law.

Wit also does not autonomously delete media, cancel queue items, expose services publicly, or weaken service authentication. Contributions that cross these boundaries are outside project scope.
